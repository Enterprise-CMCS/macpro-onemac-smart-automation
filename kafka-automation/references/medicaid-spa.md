The following page outlines the messages exchanges for a SPA from the source. 

# Inbound Messages 
These are messages generated by OneMAC system and published to `aws.onemac.migration.cdc.devtest` or `aws.onemac.migration.cdc.qatest` kafka topics.

# Outbound Messages
These are messages generated by OneMAC SMART app in Salesforce and published to `aws.mulesoft.onemac.events.devtest` or `aws.mulesoft.onemac.events.qatest`. The actual topic is determined based on the environment (dev or qa) being used. 
The basic JSON structure of those messages is as below:

```json
{

  "packageId": "string",
  "smartId": "string",

  "seatoolStatus": "string",

  "proposedDate": "dateType",
  "approvedEffectiveDate": "dateType",
  "finalDispositionDate": "dateType",
  "changedDate": "dateType",
  "statusDate": "dateType",
  "raiRequestedDate": "dateType",
  "raiWithdrawnDate": "dateType",


  "subject": "string",
  "description": "string",


  "types": [
    {
      "typeId": "string",
      "typeName": "string"
    }
  ],
  "subTypes": [
    {
      "typeId": "string",
      "typeName": "string"
    }
  ],


  "cpoc": {
    "firstName": "string",
    "lastName": "string",
    "email": "string"
  },
  "reviewTeam": [
    {
      "firstName": "string",
      "lastName": "string",
      "email": "string"
    }
  ]
}

```
The Field Explanations is below:

* packageId — The SPA or Waiver identifier. Note: admin change events use packageId instead of id  — confirm standardization with the team.
* smartId — The Salesforce/SMART correlation identifier used to link the OneMAC record to the SMART record. This is same as the salesforceId  we talked earlier. 
* seatoolStatus  — The status of the object in SMART.  
* proposedDate — Filled by the state, but CMS reviewers have the ability to override this date.
* approvedEffectiveDate — The effective date of approval, sourced from SEATool/SMART.
* finalDispositionDate — The date of adjudication (final decision), sourced from SEATool/SMART.
* changedDate — A new SMART field (per Erika) that also exists in the main index alongside makoChangedDate. Used to handle out-of-order and stale message filtering.
* statusDate — A new SMART field (per Erika) that also exists in the main index. Tracks when the status last changed.
* raiRequestedDate — The date CMS issued the RAI. Defaults to today's date, but in SEATool a reviewer can provide a specific (backdated) date.
* raiWithdrawnDate — The date the state withdrew their submission after an RAI was issued.
* subject — The subject (aka state plan title) 
* description — The description of the submission package (a.k.a state plan summary memo) 
* types — An array of type objects. Each has a typeId (moving to string going forward — discuss whether number can still be supported) and a typeName.
* subTypes — An array of subtype objects. Same structure as types — typeId and typeName. Confirm number vs string alignment with the team.
* cpoc — The lead analyst 
* reviewTeam — An array of SRT (State Review Team) members, each with firstName, lastName, and email.
