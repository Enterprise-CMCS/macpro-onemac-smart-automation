# Guidelines to generate test scenarios. 

## Folder Structure
1. The scenarios are stored in sub-folders of `kafka-automation/scenarios`.
2. A scenario folder naming convention is `s<number>` (example `s002`).
3. There will be a `scenario.yml` that stores the metadata. 
4. There will be additional JSON files that contains the payload (example 1.json, 2.json etc). 

## scenario.yml structure
The `scenario.yml` will have the metadata about the scenario and the fields are:
```
scenario_id: This will be the folder
id: This `id` or `packageId` found in the payload
description: A small description of the scenario and provides a human readable overview. 
delay: This is a delay in seconds, that must be applied before publishing messages. 
files: This is a list of payload files. 
```

## References 
`medicaid-spa.md` file have the examples available that can be tweaked to generate payloads for scenarios.

## Tips For Generating Scenario payloads
- Every scenario folder will be for a single SPA. So ensure that the event payloads all have the same SPA and package identifiers. 
- The `id` in inbound samples and packageId in outbound samples represents SPA ID. (Example: "id": "ZZ-26-0001")
- The generation process must make sure that it auto increments the next SPA ID (Example: "ZZ-26-0002" when "ZZ-26-0001" is in use. )
- The `timestamp` field most of the time represents the current time in milliseconds. 
- In the attachments, `uploadDate` represents the upload time in milliseconds.
- The `proposedEffectiveDate` also represents time in milliseconds, but will be for a future date, may be one or two months into future. 
- The `smartId` in outbound message is generated by Salesforce and will be having a one-to-one correlation with SPA. 

## Inventory
The `inventory.md` maintains a table that outlines the scenarios that we have generated.

## Generation Workflow

Follow these steps when creating a new scenario:

### 1. Gather Context
- Read `kafka-automation/references/medicaid-spa.md` for message templates (inbound and outbound).
- Read `kafka-automation/scenarios/inventory.md` to find the next available scenario number and SPA ID.
- Read an existing scenario folder (e.g., `kafka-automation/scenarios/s002/`) to understand the structure.

### 2. Determine IDs
- **Scenario ID** — check `inventory.md` for the highest `s###` number and increment.
- **SPA ID** — if the user provided a SPA ID, use it exactly as given. Otherwise, ask the user: "Would you like to provide a SPA ID or should I auto-generate one?" If auto-generating, check `inventory.md` for the highest `ZZ-26-####` and increment. Either way, verify the chosen SPA ID is not already in use in `inventory.md`.

### 3. Ask the User (if needed)
If the user hasn't specified what events the scenario should include (new submission, subsequent documents, withdrawal, etc.), ask before generating.

### 4. Generate Payload Files
Create `1.json`, `2.json`, … using templates from `medicaid-spa.md`:
- All payloads in a scenario share the same SPA ID.
- Set `timestamp` and `uploadDate` to current time in milliseconds.
- Set `proposedEffectiveDate` to 1–2 months in the future (milliseconds).
- Use unique UUIDs for attachment `key` fields.

### 5. Create scenario.yml
Include `scenario_id`, `id`, `description`, `delay` (default 5), and `files` list.

### 6. Update Inventory
Append a row to `kafka-automation/scenarios/inventory.md` with the new scenario ID, SPA ID, and overview.

### 7. Summarize
Report what was created: scenario ID, SPA ID, number of events, and a brief description.

## Constraints
- DO NOT modify existing scenario files or payloads.
- DO NOT create scenarios with SPA IDs that are already in use (check `inventory.md`).
- DO NOT invent new event types — only use events documented in `medicaid-spa.md`.
- ONLY generate files inside `kafka-automation/scenarios/`.
- ALWAYS update `kafka-automation/scenarios/inventory.md` after creating a scenario.

## Deleting a Scenario
When asked to delete a scenario:
1. Remove the scenario folder and all its contents (e.g., `kafka-automation/scenarios/s003/`).
2. Remove the corresponding row from `kafka-automation/scenarios/inventory.md`.
3. Confirm what was deleted.
